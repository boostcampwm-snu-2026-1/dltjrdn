# 설계 결정 (Design Decisions)

왜 이렇게 만들었는지 — 졸업작품 기여 포인트와 직결되는 선택들.

## 1. 역할 분리 — Gemini vs SDXL
- **Gemini** = 입력 이해 + 프롬프트 설계 (텍스트·이미지 → 영어 프롬프트).
- **SDXL** = 스킨 픽셀 생성 (프롬프트 → 64×64 이미지).
- 이유: 이미지 이해/언어는 멀티모달 LLM이, 스킨 픽셀은 스킨으로 파인튜닝된 확산 모델이 잘함. 한쪽에 다 시키지 않는다.
- **기여 포인트**: ① 프롬프트 엔지니어링 + 파이프라인 설계. (단순 API 연결로 보이지 않게 이 부분을 강조)

## 2. 프롬프트 규칙 (`prompt_skill.py`)
- 영어 한 줄, **캐릭터** 묘사(장면 X), 8~20단어.
- 포함: 종류/아키타입, 의상, **색상**, 헤드기어, 액세서리 1~3개.
- **고유명사(캐릭터·작품 이름) 출력 금지** → 외형으로 번역.
  - 근거: SDXL은 "Naruto" 같은 이름보다 "orange jumpsuit ninja" 같은 외형을 잘 그림. OC·무명 캐릭터에서도 동작. 저작권상도 안전.
- 금지: 포즈/앵글/배경/조명/화질 표현, 단어 `Minecraft`/`skin`/`pixel art`.

## 3. 특정 캐릭터는 "텍스트"보다 "이미지"
- Gemini는 마이너 캐릭터(예: 크롱)를 **이름으로는** 잘 못 알아봄 → 텍스트만 주면 엉뚱하게(개구리 등) 묘사.
- 해결: **이미지 업로드 모드** — Gemini가 실제 생김새를 보고 묘사하므로 정확.
- 교훈: "이름 인식"에 기대지 않고 "외형 관찰"에 기대는 설계가 견고하다.
- (구글 검색 grounding도 시도했으나, 무료 한도를 크게 깎아서 제거함. → [git.md] 한도 이슈)

## 4. 스킨 형식 불변 규칙
- 출력은 **64×64 RGBA PNG** (마인크래프트 규격, 협상 불가).
- UV 좌표는 `mc_skin_generator.py`의 `BASE_LAYER_RECTS`/`OVERLAY_LAYER_RECTS`가 **단일 출처**.
- 베이스 레이어 면 = 완전 불투명(alpha 255), 겉 레이어 = 투명 허용.
- 모든 생성 결과는 `validate_skin()` 통과(빈 리스트) 필수.
- 64×64는 "저해상도"가 아니라 규격. 화면에선 3D/전개도를 확대해 보여줄 뿐.

## 5. SDXL 후처리 (`sdxl_skin.py`)
- 모델: `monadical-labs/minecraft-skin-generator-sdxl`.
- 원본 추론 방식을 따름:
  1. **768×768** 생성 (모델 학습 크기)
  2. **상단 절반** 크롭 (스킨은 위쪽 절반에 있음)
  3. **64×32** 레거시 스킨으로 축소(NEAREST)
  4. **64×64** 모던 포맷으로 변환 (왼팔/왼다리 = 오른쪽 복사)
  5. 겉 레이어 투명 처리
- 추론 스텝 50, guidance 7.5.
- 흔한 실수: 768을 통째로 64×64로 줄이면 아래쪽 쓰레기 영역까지 입혀져 깨진다. (실제로 겪음)

## 6. 로컬 GPU 선택 (vs Colab)
- RTX 4060 Ti(8GB) 보유 → 로컬 채택.
- 근거: 웹앱이라 백엔드에서 바로 모델 호출 가능(터널 불필요), 세션 안 끊김, 무료 Colab(T4)보다 빠름.
- 8GB 대응: fp16 + `enable_model_cpu_offload` + attention slicing + VAE tiling.

## 7. 직접 프롬프트 모드 (임시)
- `direct=true`면 Gemini를 거치지 않고 입력 영어 텍스트를 프롬프트로 직행.
- 목적: Gemini 무료 한도가 빡빡해 ② SDXL을 빠르게 반복 테스트하기 위함.
- 임시 기능 — 한도 해결 후 기본 Gemini 경로로 복귀 예정.

## 알려진 이슈
- 몸통이 마젠타(투명 마커)로 나오는 경우 → 배경 투명 후처리 보강 필요.
- Gemini 무료 한도 매우 제한적(2.5-flash 일 20, 2.0-flash 0) → 본격 개발 시 결제 등록 고려.
